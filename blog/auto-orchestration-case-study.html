How We Built an OS with Auto-Orchestration

A deep dive into the closed-loop orchestration pattern behind Realm OS — circuit breakers, heartbeat drift detection, self-healing services, and why we stopped writing restart scripts.

The problem with manual orchestration

Every distributed system eventually needs a supervisor. When you have 8+ services — a Rust kernel, Deno runtime, Redis bridge, AI drivers, signal bus, memory layer — something will crash. Something will drift. Something will silently fail at 3am.

The traditional answer is a restart script. Maybe a systemd unit. Maybe a Kubernetes deployment with liveness probes. All of these are reactive: they detect failure after it happens, then apply a blunt fix (restart everything).

We wanted something different for Realm OS: a system that detects drift before failure, applies targeted corrections, and learns from each incident.

The closed-loop pattern

Realm's auto-orchestration runs as a continuous loop with four stages:

1. Observe

Every service emits heartbeat signals on the signal bus at a configurable interval (default: 5s). The orchestrator tracks:

  • Last heartbeat timestamp — is the service alive?
  • Heartbeat jitter — is the interval drifting? A service under memory pressure will show increasing jitter before it crashes.
  • Error rate — are errors spiking relative to the baseline?
  • Dependency health — are the services this one depends on healthy?
heartbeat { service: "nexus", ts: 1707350400, jitter_ms: 12, errors_1m: 0 }
heartbeat { service: "redis_bridge", ts: 1707350400, jitter_ms: 3, errors_1m: 0 }
heartbeat { service: "mana_vault", ts: 1707350400, jitter_ms: 8, errors_1m: 2 }

2. Orient

The orchestrator compares observations against known-good baselines. Each service has a health profile:

  • Green: All metrics within normal range
  • Yellow: Jitter > 2x baseline OR error rate > 3x baseline
  • Red: Heartbeat missing > 3 intervals OR error rate > 10x baseline

Yellow triggers a warning signal. Red triggers the circuit breaker.

3. Decide (Circuit Breaker)

The circuit breaker is a three-state machine:

  • Closed (normal): Traffic flows normally. All health checks pass.
  • Open (tripped): The failing service is isolated. Dependent services fall back to limp mode. A cooldown timer starts.
  • Half-Open (probe): After cooldown, a single probe request is sent. If it succeeds, the breaker closes. If it fails, the breaker reopens with an extended cooldown.

This prevents cascade failures. When the Redis bridge trips, the signal bus falls back to local-only mode. When NexusManager trips, it falls back to in-memory cache. No service crash takes down the system.

4. Act

Actions are proportional to the problem:

  • Jitter drift: Adjust heartbeat interval, log warning
  • Error spike: Reduce load, enable verbose logging
  • Service down: Restart with backoff (1s, 2s, 4s, 8s, max 30s)
  • Dependency chain failure: Restart in dependency order, verify health at each step
  • Persistent failure: Escalate to operator (Slack/Discord alert via Sentinel tier)

What this looks like in practice

Here's a real sequence from our development environment:

[14:22:03] ORCHESTRATOR: heartbeat_drift service=mana_vault jitter=45ms (baseline: 8ms)
[14:22:08] ORCHESTRATOR: heartbeat_drift service=mana_vault jitter=120ms — YELLOW
[14:22:13] ORCHESTRATOR: heartbeat_missing service=mana_vault — RED
[14:22:13] CIRCUIT_BREAKER: OPEN service=mana_vault cooldown=5s
[14:22:13] MANA_VAULT: entering limp_mode (in-memory only)
[14:22:18] CIRCUIT_BREAKER: HALF_OPEN service=mana_vault — probing
[14:22:18] ORCHESTRATOR: probe_success service=mana_vault latency=3ms
[14:22:18] CIRCUIT_BREAKER: CLOSED service=mana_vault
[14:22:18] MANA_VAULT: exiting limp_mode — Redis reconnected

Total downtime: 5 seconds. No operator intervention. No restart script. The system detected drift (jitter increasing), tripped the breaker before cascade failure, waited for recovery, verified health, and resumed normal operation.

Lessons learned

1. Jitter is the canary. CPU pressure, memory leaks, and I/O contention all show up as heartbeat jitter before they cause outright failure. Monitoring jitter gives you a 5-15 second warning window.

2. Limp mode is non-negotiable. Every service must have a degraded-but-functional fallback. ManaVault falls back to in-memory tracking. NexusManager falls back to cached queries. The signal bus falls back to local delivery. Without limp mode, circuit breakers just convert one failure into another.

3. Exponential backoff with jitter. Fixed restart intervals cause thundering herd problems when multiple services fail simultaneously. Randomized backoff (base * 2^n + random(0, base)) spreads recovery load.

4. Measure everything. The orchestrator logs every state transition with timestamps, trigger conditions, and outcomes. This audit trail is what turns "it crashed" into "here's exactly what happened and why the system recovered in 5 seconds."
